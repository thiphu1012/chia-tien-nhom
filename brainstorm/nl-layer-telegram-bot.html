The "just tell the bot" layer

Turning free-text messages like "split 540k among Aya, Ben, and Chi" into confirmed expense splits — powered by Cloudflare Workers AI, gated by a Yes/No confirmation, and writing through the same math the Mini App already uses.

1 · The mental model

The model is a parser / router, never the source of truth. The whole feature is four moves:

  1. Parse Worker sends the message text + the event's participant list + the split_expense tool definition to Workers AI. It returns one tool call with filled-in arguments (amount, who's included, weights) — or nothing if it didn't understand.
  2. Confirm Worker replies with an inline ✅ Yes / ❌ No keyboard summarizing the parsed split. No money is written yet.
  3. Execute Only when the user taps Yes, the Worker runs the same pure DB operation the Mini App uses. Settlement math stays in settle.ts.
  4. Reply Worker posts the result back into the chat.
🔒 Non-negotiable invariants (from CLAUDE.md)
  • Identity is never from the model. The payer / acting user is update.message.from, authenticated by Telegram via your webhook secret. The model may fill amount & members, never who.
  • Money stays integer đồng. The model normalizes 540k → 540000; no float math ever touches balances.
  • Every money mutation is confirmed before the write.

2 · Where the "LLM" lives

No model runs on your server. Your Cloudflare Worker stays a thin orchestrator and makes one call to the on-platform env.AI binding at request time.

Runtime call

Workers AI function-calling contract — note this is not Anthropic's shape.

const r = await env.AI.run(env.AI_MODEL, {
  messages: [
    { role: "system", content: SYSTEM_PROMPT },
    { role: "user",   content: text },
  ],
  tools: [SPLIT_EXPENSE_TOOL],
});
// read: r.tool_calls?.[0]  → { name, arguments }

Dev vs Prod (orthogonal to model)

  • Dev — wrangler dev, [ai] binding = "AI" in wrangler.toml, POST it fake Telegram updates. DEV_MODE only affects the Mini App auth bypass, not the bot.
  • Prod — npm run deploy, same code. Workers AI needs no external key; it's billed via your Cloudflare account.

The model is a one-line string behind the binding — plan it as an AI_MODEL var so you can swap or A/B it with no code change.

call / request return / response background note
💡 Why the two phases are separate updates

A Workers AI round-trip takes seconds, but Telegram retries the webhook if it doesn't 200 fast — so phase ① acks immediately (m3) and does the parse in ctx.waitUntil. That means threading ctx: ExecutionContext into fetch (it isn't passed today, src/index.ts). And the Yes tap arrives as a separate callback_query update (m12) — which is exactly why the parsed action is stashed in a pending_actions D1 row (m8) and referenced by a short id in the button's callback_data (64-byte limit).

🇻🇳 Accuracy lever that matters more than model size

These are general-purpose models. Parse quality on Vietnamese/mixed input (e.g. "chia 540k cho Aya với Ben tính đôi") depends heavily on a well-written system prompt: Vietnamese few-shot examples, explicit k → ×1000 and tr → ×1,000,000 đồng normalization, and a clear instruction that weight means shares (tính đôi → weight 2). The confirm step is the safety net for whatever slips through.

5 · The one tool: split_expense

Few, goal-shaped. The model fills business args only; the server resolves names against the participant list and enforces identity.

{
  name: "split_expense",
  description: "Split one shared expense among event members with optional weights.",
  parameters: {
    type: "object",
    properties: {
      title:  { type: "string",  description: "What the expense was for" },
      amount: { type: "integer", description: "Total in whole đồng (540k → 540000)" },
      members: {
        type: "array",
        description: "Participants included, by name as written",
        items: {
          type: "object",
          properties: {
            name:   { type: "string" },
            weight: { type: "integer", description: "Shares; default 1, ×2 = 2" }
          },
          required: ["name"]
        }
      }
    },
    required: ["amount", "members"]
  }
}

Server maps each name → a participants.id (fuzzy + @mention entities for real user IDs), builds the splits[] the existing writeSplits() expects, and the payer is always the verified Telegram sender.

7 · Senior review — findings & open decisions

A pass over this design as reviewed code. The architecture holds (confirm-gate, one write path, identity from Telegram); the gaps are in the state-machine edges — expired, duplicate, unauthorized, unbound, unclaimed — which is where money apps earn trust. Each is ~5 lines if decided now.

CriticalTelegram privacy mode: the "plain text" path mostly won't fire in groups

By default, bots in groups only receive commands, @mentions, and replies to the bot — not arbitrary messages. So "chia 540k cho Aya, Ben" typed plainly never reaches the webhook. Invisible in local testing (we POST fake updates straight to /webhook), guaranteed to surface in the first real group.

Decide: (A) keep privacy mode ON — users write @TallyBot chia 540k…; less noise, less cost, no "bot reads everything" concern  ·  (B) disable via BotFather — bot sees every message, must cheaply filter before parsing. Lean: A.
Proposed fix (option A): keep privacy mode ON and gate on addressed-to-bot. Add a BOT_USERNAME var to wrangler.toml; in groups, only parse when the text starts with the mention or the message replies to the bot. Private chats parse as-is. 
const mention = "@" + env.BOT_USERNAME;
const addressed = isPrivate
  || text.startsWith(mention)
  || msg.reply_to_message?.from?.username === env.BOT_USERNAME;
if (!addressed) return ok();          // stay silent in groups
const query = text.replace(mention, "").trim();  // strip before parsing

Criticalpending_actions has no lifecycle

  • TTL — a Yes tap on a week-old card must reply "hết hạn, gõ lại nhé" (e.g. 15-min expiry checked at callback time).
  • Idempotency — double-tap Yes → duplicate expense. Consume the row atomically: UPDATE … SET status='done' WHERE id=? AND status='pending', proceed only if meta.changes === 1.
  • The ❌ No branch — delete the row + edit the message; currently unspecified.
  • Wrong-user tap — anyone in the chat can press the button; non-creator gets answerCallbackQuery("không phải của bạn").
  • Every tap needs a prompt answerCallbackQuery, or Telegram shows a ~30s spinner on the button.
Proposed fix: give the row an explicit state machine — pending → done | cancelled | expired — and make Yes a single atomic transition. 
CREATE TABLE pending_actions (
  id         TEXT PRIMARY KEY,      -- short id → callback_data "t:yes:<id>"
  chat_id    INTEGER NOT NULL,
  user_id    INTEGER NOT NULL,      -- only this user may confirm
  event_id   TEXT NOT NULL,
  tool       TEXT NOT NULL,         -- 'split_expense'
  args_json  TEXT NOT NULL,         -- resolved participant IDs (see Risk)
  status     TEXT NOT NULL DEFAULT 'pending',
  created_at INTEGER NOT NULL
);

CriticalSender may have no participant row → the write 403s

src/api.ts requires the actor to be a claimed participant (the mine check) before adding an expense. A group member who never opened the Mini App fails it — the first thing a real group hits.

Decide: (A) auto-create + claim a participant from their Telegram first_name at confirm time — smoothest  ·  (B) reject with "mở Tally để tham gia trước". Lean: A.
Proposed fix (option A): an ensureParticipant() helper, called at execute time (after ✅ — confirm stays side-effect-free). Try, in order: already claimed → claim an unclaimed row with the same normalized name → create fresh. 
async function ensureParticipant(env, eventId, u /* TgUser */) {
  const mine = await one(`… WHERE event_id=?1 AND user_id=?2`, eventId, u.id);
  if (mine) return mine.id;
  const unclaimed = await one(          // "Phú" added by hand in the Mini App
    `… WHERE event_id=?1 AND user_id IS NULL AND norm(name)=?2`,
    eventId, norm(u.first_name));
  if (unclaimed) { await claim(unclaimed.id, u.id); return unclaimed.id; }
  const id = uid();                     // brand-new member
  await insertParticipant(id, eventId, u.first_name, u.id);
  return id;
}
The middle branch matters: groups typically pre-add members by name in the Mini App, so the sender usually exists — just unclaimed. Claiming (not duplicating) keeps the settlement table clean.

Critical/tally binds "the newest event" — of whom?

The user running /tally may have several events; silently binding their newest is a misfire waiting to happen.

Fix: /tally replies with the user's recent events as inline buttons → tap one to bind this chat. Reuses the same inline-keyboard plumbing the confirm step already needs.
Proposed fix: picker + exactly one bound event per chat, so resolution is a single unambiguous row (no "newest wins" heuristic at all).
  • /tally → SELECT the sender's 5 most recent events → inline keyboard, one button per event (callback_data "t:bind:<eventId>"). No events → reply guiding them to create one in the Mini App first.
  • On tap, bind atomically — unbind everything else in this chat, then bind the chosen one:
await env.DB.batch([
  stmt(`UPDATE events SET chat_id=NULL WHERE chat_id=?1`, chatId),
  stmt(`UPDATE events SET chat_id=?1  WHERE id=?2`, chatId, eventId),
]);
// resolution everywhere else becomes:
// SELECT * FROM events WHERE chat_id=?1   → 0 or 1 row, never a guess
Re-running /tally later is the "switch active event" gesture — no extra command needed.

CriticalFail closed on WEBHOOK_SECRET

src/telegram.ts:15 skips verification when the secret is unset. Tolerable when the webhook only sent help text; once callback_query executes money writes, an unset secret means anyone who finds the URL can forge updates. Make the check unconditional in this feature.

Proposed fix: a one-line flip from fail-open to fail-closed — missing secret now rejects instead of skipping the check. 
- if (env.WEBHOOK_SECRET && req.headers.get("X-Telegram-Bot-Api-Secret-Token") !== env.WEBHOOK_SECRET)
+ if (!env.WEBHOOK_SECRET || req.headers.get("X-Telegram-Bot-Api-Secret-Token") !== env.WEBHOOK_SECRET)
    return new Response("forbidden", { status: 403 });
Plus a README note: WEBHOOK_SECRET is now required — a deploy without it makes the bot refuse all updates loudly, instead of accepting forged ones silently.

RiskDecide explicitly, don't improvise later

  • Tool args are untrusted input. Validate between model output and pending_actions: amount integer > 0 with a sane upper bound; members non-empty; weights integers 1..~20; coerce strings (the model will emit "540k" occasionally).
  • Name resolution needs rules, not vibes. Normalize diacritics ("Phu" = "Phú") → exact match → unique-prefix match → otherwise refuse and ask, never guess. Unknown name = "⚠️ không tìm thấy 'X'" in the confirm card, never silently dropped.
  • Store resolved participant IDs in pending_actions, not names — and re-validate they still exist at execute time (members can change between confirm and Yes).
  • Workers AI failure path. Occasional 503s/timeouts: one retry, then a graceful "thử lại sau". Optional: put Cloudflare AI Gateway in front of the binding for retries + analytics.
Proposed fixes: all four live in src/nl.ts as small pure functions (unit-testable, no I/O).
  • Arg validation — accept integers or the Vietnamese shorthand the model may echo; anything else is a parse failure, never a guess:
function parseAmount(v: unknown): number | null {
  if (typeof v === "number" && Number.isFinite(v)) return Math.round(v);
  const m = String(v).trim().toLowerCase()
    .match(/^(\d+(?:[.,]\d+)?)\s*(k|tr|m|củ|cu)?$/);   // 540k · 1.5tr · 2m
  if (!m) return null;
  const mult = { k: 1e3, tr: 1e6, m: 1e6, "củ": 1e6, cu: 1e6 }[m[2]] ?? 1;
  const n = Math.round(parseFloat(m[1].replace(",", ".")) * mult);
  return n > 0 && n <= 10_000_000_000 ? n : null;      // 0 < amount ≤ 10 tỷ
}
// members: 1..50, dedup by resolved id
// weight:  Math.min(20, Math.max(1, Math.round(Number(w) || 1)))
  • Name resolution ladder — normalize → exact → unique prefix → refuse:
const norm = (s: string) => s.normalize("NFD")
  .replace(/\p{M}/gu, "").replaceAll("đ", "d").replaceAll("Đ", "d")
  .toLowerCase().trim();                       // "Phú" → "phu"

function resolveName(input: string, ps: {id:string; name:string}[]) {
  const q = norm(input);
  const exact  = ps.filter(p => norm(p.name) === q);
  if (exact.length === 1)  return { id: exact[0].id };
  const prefix = ps.filter(p => norm(p.name).startsWith(q));
  if (prefix.length === 1) return { id: prefix[0].id };
  return { error: exact.length + prefix.length === 0 ? "unknown" : "ambiguous" };
}
// ANY unresolved name → no pending row; bot replies
// "⚠️ Không tìm thấy 'X' — thành viên gồm: Aya, Ben, Chi…"
  • IDs in the payload — args_json stores {participantId, weight}[]; at execute time, SELECT those ids for the event and expire the card if any vanished.
  • AI resilience — wrap env.AI.run: 10s timeout, one retry, then reply "Bot đang bận, thử lại sau nhé 🙏". AI Gateway ([ai] gateway in wrangler.toml) adds retries/analytics with zero code if wanted.
