The open question — which model reads the receipt

Original plan was OCR model → LLM. A modern vision LLM does OCR + item/price/total extraction in one call and reads layout, not just characters — which is exactly where a crumpled Vietnamese receipt with a service-charge line beats a two-stage pipeline. So the real choice is: which vision model, and can it stay free.

Headline finding: the free tier is not the constraint

Cloudflare Workers AI gives 10,000 Neurons/day free (no card), then $0.011 / 1,000 Neurons. A single receipt scan costs on the order of 45–95 Neurons — so the free tier buys roughly 100–220 scans per day, far more than a shared-expense bot needs. Since every Workers AI vision model fits your volume, the decision becomes accuracy on messy VN receipts, not price.

Recommendation

Start here   Ship the MVP on Workers AI — Llama 4 Scout 17B (or Llama 3.2 11B Vision if you want max free volume while prototyping). Fully on-platform, no external key, no egress, comfortably inside the free tier.

Design one seam so the model is swappable. Put parsing behind a single parseReceipt(imageBytes) → {items, charges, total} function. If real Vietnamese receipts show accuracy problems (missed items, mangled 540.000, wrong totals), flip that one function to Claude Haiku 4.5 via Cloudflare AI Gateway — still Cloudflare-fronted (caching, rate limits, observability), just a paid backend. Decide with a small labelled set of your own receipts, not in the abstract.

Risks to design around (model-agnostic)

1 · Vietnamese number formatting is a 1000× trap

540.000 is 540000 đồng, not 540 — the dot is a thousands separator. Your hard invariant is integer đồng, so a parser that reads that dot as a decimal invents a 1000× error. The extraction prompt must return raw integer đồng, and you should validate the sum before writing anything.

2 · Structured output is less reliable than Claude's tool-calling

Workers AI vision models don't do function-calling as cleanly as Claude. Prompt for strict JSON (Workers AI supports response_format / JSON mode on some models), then parse + validate + retry once on malformed output. Don't trust the first response blindly.

3 · Totals don't reconcile — VAT, service charge, discounts

Real bills carry phí phục vụ (service), VAT, and rounding, so Σ items ≠ total. Decide a rule: distribute charges proportionally across items, or add them as one shared line everyone's in. Validate Σ parsed items + charges == parsed total and surface a mismatch in the Mini App review screen rather than silently trusting it.
